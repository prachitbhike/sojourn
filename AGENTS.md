# Sojourn — agent conventions

This file is the contract every coding-agent thread (Claude Code, Codex, etc.) inherits when working in this repo. Read it before doing anything else. It points at the canonical specs and lays out the rules that keep parallel slices from stomping each other.

## What this repo is

A web app for creating animatable **pixel-art** sprites with a chat/inspector-driven UX. Sprite poses + animations are generated by **PixelLab**; character portraits and reference fusion go through **nano banana** (Gemini 2.5 Flash Image); voice runs through **ElevenLabs**. Frame-based animation (sprite sheets), not rigged.

## Source-of-truth docs

Read these before writing code. If your task contradicts them, **stop and flag it** — don't silently diverge.

- `docs/phase-0-plan.md` — the current phase spec. Stack choices, data model, API surface, URL strategy, stub asset catalog, deliverables checklist.
- `docs/decisions.md` — running log of consequential decisions and why. Append to it when a new decision surfaces that wasn't in the plan.
- `docs/execution-plan.md` — slicing for Phase 0, the prompt each slice's thread should be started with, the per-slice review checklist, and what to do when a slice goes sideways. Read your slice's prompt before starting work.

## Stack at a glance

- **Frontend:** Vite + React + TS, Phaser sandboxed inside one `<SpriteStage>` component
- **API:** Node + Hono + TS
- **DB:** SQLite via Drizzle (Turso in prod, local file in dev)
- **Storage:** Cloudflare R2 (S3-compatible)
- **Generators:** PixelLab (sprite poses + animations), nano banana / Gemini 2.5 Flash Image (portraits + reference fusion). Both behind a `Generator` interface in `packages/shared/generators` — see D08, D09 in `docs/decisions.md`.
- **Voice:** ElevenLabs (Phase 2+)
- **Hosting:** Vercel (frontend) + Fly/Railway (API) — *specific choice deferred*

## How work is structured

Phase 0 is sliced into 5 PRs (see `docs/phase-0-plan.md`). Each thread owns one slice. The slicing exists so reviews are tractable — don't merge across slice boundaries.

**Hard rules for every thread:**

1. **Stay in your slice.** If you notice something out-of-scope worth fixing, mention it in the PR description and stop. Don't refactor adjacent code.
2. **Don't change the data model or URL scheme** without explicit approval. These are expensive to revisit once shared characters exist.
3. **Justify every new dependency.** If it's not in `docs/phase-0-plan.md`, the PR description must explain why it's needed and what alternatives were considered.
4. **One PR per slice.** Open it when the slice's acceptance criteria pass. Don't bundle slices.
5. **Update `docs/decisions.md`** when you make a call that wasn't pre-decided in the plan. One short entry, format documented in the file.

## Code conventions

- TypeScript strict mode everywhere
- No default exports (named only) — easier to grep, easier to refactor
- Drizzle schema lives in `packages/shared/schema.ts`; both apps import from there
- API request/response shapes live in `packages/shared/contracts.ts`; never duplicate types between apps
- Server-side: `pino` for logs, never `console.log`
- Client-side: `console.error` is fine; no analytics yet

## Parallel work hygiene

Slices 2 and 3 are designed to run in parallel. If you're one of those threads:

- You're working in a git worktree — your branch is isolated, but `pnpm-lock.yaml` conflicts are still possible. Avoid adding deps unless you must.
- Slice 2 owns `apps/api/**` exclusively. Slice 3 owns `apps/web/src/components/SpriteStage/**` and a demo route. Anything in `packages/shared/**` is shared — coordinate via the plan doc, don't freelance.

## Things to watch for (recurring footguns)

- **Stub assets must round-trip through the real load path.** Don't special-case stubs in the renderer — they should look identical to real nano banana output to the rendering code, just smaller/uglier. Otherwise Phase 1 surfaces a bunch of integration bugs.
- **`SameSite=None; Secure` cookies break in localhost.** Use `SameSite=Lax` in dev, switch in prod via env. Documented in `docs/phase-0-plan.md` infra section.
- **Edit-key in URL** is the security model in Phase 0. Don't add account-style auth checks; they'll conflict with the rotation flow.
- **Phaser is sandboxed.** No Phaser imports outside `apps/web/src/components/SpriteStage/`. The rest of the app talks to it through the imperative handle only.
- **Generators are sandboxed too.** No PixelLab / nano banana / ElevenLabs SDK imports outside `packages/shared/generators/<provider>/`. API handlers dispatch through the `Generator` interface only — never touch a provider SDK directly. This is what makes swapping providers a contained change.
- **Don't conflate portraits and sprites.** Portraits are illustrated, single-image, served as plain `<img>`, generated by nano banana. Sprites are pixel-art, sprite-sheet + manifest, rendered through Phaser, generated by PixelLab. Different fields on the schema, different endpoints, different generators.
- **Don't run `git add -A` or `git add .`.** `.env.local` and the keys file at repo root must never be committed. Stage files explicitly by path.

## When to stop and ask

- The slice's acceptance criteria can't be satisfied without changing the plan
- You hit an architectural decision the plan doesn't cover
- A test is failing in a way that suggests a real bug in another slice's code
- The plan and an existing decision in `decisions.md` contradict

In these cases: leave a clear summary in the PR description, mark it draft, and surface it to the human reviewer. Don't push through.
