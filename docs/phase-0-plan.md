# Phase 0 — Shareable skeleton with stub generation

The point of Phase 0 is to validate the **data model, URL strategy, share/edit flow, and Phaser rendering loop** before any real API calls. All generation endpoints return canned placeholder sprites. If the share-URL flow works end-to-end with a fake character, swapping in nano banana in Phase 1 becomes a contained change.

## Stack lock-in

| Layer | Choice | Why |
|---|---|---|
| Frontend | Vite + React + TS + Phaser | Already validated; "shareable" doesn't change this |
| Server | Node + Hono (or Express) | Persistent WebSocket later for ElevenLabs streaming → rules out edge-only platforms |
| DB | SQLite via Drizzle (Turso for prod) | Simple, free tier, works locally and in prod with the same code |
| Asset storage | Cloudflare R2 | Cheapest egress for shared assets; S3-compatible API |
| Sprite generation | **PixelLab** (Phase 1+) | Purpose-built for pixel-art sprites; native animation + pose primitives; stub-only in Phase 0 |
| Portraits / refs | **Nano banana** (Gemini 2.5 Flash Image) (Phase 1+) | High-fidelity static portraits + multi-image reference fusion for character design; not used for sprites |
| Voice | ElevenLabs (Phase 2+) | TTS + streaming + alignment for lipsync |
| Hosting | Vercel (frontend) + Fly.io or Railway (Node API) | Vercel is ideal for the React app; the Node server lives separately so it can hold WebSockets |

**Aesthetic:** pixel art (see D07). The generator split (PixelLab for sprites, nano banana for portraits) is per D08, behind the generator abstraction in D09.

## URL & share model

```
/                          → landing / "create new" entry point
/c/:slug                   → public read-only viewer (anyone can see + play)
/c/:slug/edit?key=:editKey → editor (full inspector, generation, voice)
```

- `slug` = nanoid(8) + optional `-name-kebab` suffix (e.g. `/c/aB3kF9xQ-blue-wizard`)
- `editKey` = nanoid(24), generated on creation, stored as a cookie *and* embedded in the URL the creator sees
- No accounts in Phase 0. Lose the edit URL = lose edit access. Phase 3 can layer accounts on top by binding `editKey` → user.
- Public read works without cookies, edit key, or auth. This is what makes it "shareable from day one."

## Data model

```ts
// drizzle schema (sketch)
characters: {
  id: text (nanoid pk)
  slug: text (unique idx)
  editKeyHash: text             // hash, never store raw
  name: text
  basePrompt: text              // user's freeform description
  refImageUrl: text | null      // R2 URL if user uploaded a reference
  attributes: jsonb             // { archetype, outfit, palette, expression, ... }
  portraitUrl: text | null      // canonical portrait (illustrated, from nano banana)
  portraitGenerator: enum       // "stub" | "nano-banana"
  voiceId: text | null          // ElevenLabs voice id
  createdAt, updatedAt
}

poses: {
  id, characterId (fk → characters.id, ON DELETE CASCADE)
  name: text                    // "idle", "walk", "attack"
  spriteSheetUrl: text          // R2 URL (pixel art, from PixelLab)
  manifest: jsonb               // { frameWidth, frameHeight, frames: [{x,y,duration}] }
  generator: enum               // "stub" | "pixellab"
  status: enum                  // "pending" | "ready" | "failed"
}
```

Two separate artifact types, two separate generators (per D08):
- **Portrait** (`portraitUrl`) — one illustrated image per character, surfaces in the inspector, share cards, public viewer header. Generated by nano banana.
- **Sprite poses** (`poses[].spriteSheetUrl`) — pixel-art sprite sheets, played on the Phaser stage. Generated by PixelLab.

The `portraitGenerator` and `generator` fields exist so Phase 0 stub assets (`'stub'`) and Phase 1+ real assets coexist without schema churn. The dispatch lives in `packages/shared/generators` per D09.

Indexes: `slug` unique, `(characterId, name)` unique on poses (one canonical idle/walk per character). Cascade-delete poses when a character is deleted so we never leak orphaned R2 objects via stale rows.

Phaser loads `spriteSheetUrl` + `manifest` directly — keep the manifest shape Phaser-native (`frameWidth`, `frameHeight`, frame array) so there's no translation layer.

## API surface (all stubbed in Phase 0)

```
POST   /api/characters                    { prompt?, refImage? } → { slug, editKey }
GET    /api/characters/:slug              → public character + portrait + poses
PATCH  /api/characters/:slug              (auth: editKey) → updates attributes (no auto-regen)
POST   /api/characters/:slug/portrait     (auth: editKey) → 202, queues portrait gen (nano banana / stub)
POST   /api/characters/:slug/poses        (auth: editKey) { name } → 202, queues sprite gen (pixellab / stub)
POST   /api/characters/:slug/voice        (auth: editKey) { text } → audio URL
POST   /api/characters/:slug/rotate-key   (auth: current editKey) → { editKey } (new)
```

Portrait and sprite generation are separate endpoints because they hit different providers, have different cost profiles, and fail independently. `PATCH` only updates DB rows — re-generation is always an explicit call, so the UI controls when to spend on a re-render.

`rotate-key` is for the "I leaked my edit URL" case: hits the same auth as any other write, mints a new `editKey`, replaces `editKeyHash`, invalidates the old. Endpoint exists in Phase 0 (it's part of validating the edit-key model); the UI affordance can land in Phase 1.

In Phase 0, both `/portrait` and `/poses` dispatch to the `'stub'` generator — `/portrait` returns a baked-in illustrated PNG, `/poses` returns a hardcoded sprite sheet keyed off `name`. `POST /voice` returns a 2-second pre-recorded mp3. This is enough to drive the whole UI without any third-party calls.

## Stub asset catalog

Stubs are a load-bearing dependency for Phase 0 — spec them up front so the rest of the work has something deterministic to point at. They also become the offline-dev fallback when PixelLab / nano banana / ElevenLabs are unreachable in Phase 1+.

The catalog reflects the two-generator split: portraits are illustrated (representing nano banana output), sprites are pixel art (representing PixelLab output). Same character design across both so the visual continuity test is exercised by stubs.

| Asset | Filename | Dimensions | Aesthetic | Notes |
|---|---|---|---|---|
| Portrait | `stubs/v1/portrait.png` | 512×512, transparent PNG | Illustrated | Single static image, used in inspector + share cards |
| Idle sprite sheet | `stubs/v1/idle.png` | 64×64 frame, 4 frames horizontal (256×64) | Pixel art | 250 ms/frame, loop |
| Walk sprite sheet | `stubs/v1/walk.png` | 64×64 frame, 8 frames horizontal (512×64) | Pixel art | 100 ms/frame, loop |
| Attack sprite sheet | `stubs/v1/attack.png` | 64×64 frame, 6 frames horizontal (384×64) | Pixel art | 80 ms/frame, no loop |
| Cast sprite sheet | `stubs/v1/cast.png` | 64×64 frame, 6 frames horizontal (384×64) | Pixel art | 100 ms/frame, no loop |
| Voice clip | `stubs/v1/voice.mp3` | ~2 s, 22.05 kHz mono | — | Generic line, used for any text |

Storage:
- **Local fixtures** at `apps/api/fixtures/stubs/v1/` — committed to repo, used in dev and tests
- **R2** at `s3://sojourn-assets/stubs/v1/` — same files, used in deployed environments
- Switch via `STUB_SOURCE=local|r2` env var. Local is the default in dev so the app boots with no R2 credentials.

Each stub ships with a hand-written `manifest.json` matching the Phaser-native shape so we exercise the real load path, not a special case. The portrait stub does *not* go through Phaser — it's a plain `<img>` in the inspector / share cards.

## UI shell

Single editor page, three regions:

1. **Stage (center)** — Phaser canvas. Shows base sprite + plays current pose loop. Pose dropdown above it.
2. **Inspector (right)** — fields for `name`, `archetype` (select), `outfit` (text), `palette` (color chips), `expression` (chips), `voice` (placeholder dropdown). Each field debounced-saves and triggers a re-gen of the base sprite.
3. **Pose grid (bottom)** — cards for each pose. "+ Add pose" opens a small picker. Each card shows status (pending / ready / failed) and a thumbnail.
4. **AI-assist field** (single input, top-right of inspector) — "describe a change…" — stub in Phase 0, just appends to a transcript. Wires up real LLM in Phase 2+.

Public viewer page (`/c/:slug`) is the same Phaser stage + pose dropdown + a play button for voice — no inspector, no edit affordances.

## Concrete deliverables to call Phase 0 done

- [ ] `pnpm dev` boots Vite frontend + Node API + local SQLite, hot reloads both
- [ ] Hitting `/` and entering text creates a character, redirects to `/c/:slug/edit?key=…`
- [ ] Editor renders a stub sprite on the Phaser stage, plays an idle loop
- [ ] Editing inspector fields updates DB and re-fetches a (still stub) base sprite
- [ ] Adding a pose ("walk") shows it in the grid and plays it on the stage
- [ ] Copying `/c/:slug` to incognito → shows public viewer with stub assets, no edit UI
- [ ] R2 is wired (even if stub assets are uploaded once and reused) — proves the upload path works before nano banana lands
- [ ] Drizzle migrations, env var loading, and a basic Hono auth middleware that checks `editKey` against `editKeyHash`
- [ ] One Playwright happy-path test that scripts the deliverables above (create → edit → add pose → public view in a fresh context). This is the regression net for Phase 1.
- [ ] `pino` set up server-side, with structured logs on every auth check (success + failure, including failed `editKey` attempts with hashed prefix only). No external observability stack — stdout is fine.
- [ ] `Generator` interface in `packages/shared/generators` with two implementations: `stub` (returns catalog assets) for both portraits and sprites. Real `pixellab` and `nano-banana` implementations land in Phase 1 — Phase 0 only needs the seam. API handlers dispatch via this interface; no provider-specific code in handlers.

## Decisions, push back if any are wrong

1. **No accounts in Phase 0** — edit-key-in-URL model. Simpler, ships faster. Risk: someone shares the wrong URL once and loses control. Mitigation: show a clear "this is the edit URL — keep it private" banner on first creation.
2. **R2 over Vercel Blob** — R2 is cheaper at scale and you're not locked to Vercel. Costs you ~30 min of S3 SDK setup vs. Blob's drop-in.
3. **Hono over Express** — smaller, TS-first, runs the same on Node and Bun, easier WebSocket later. If you have a strong Express preference, switch.
4. **Turso over Postgres** — for a single-user-creation prototype, SQLite is plenty and Turso gives you replication if you need it. Switch to Neon later if relations grow.
5. **Phaser as the renderer, but rendered as a *component*** — wrap Phaser in a single `<SpriteStage>` React component, expose `setPose(name)` / `play()` as imperative handles. Keeps Phaser sandboxed; rest of app is plain React.

## Infra glue

Cross-origin and env scaffolding that's painful to retrofit:

**Local dev (same-origin via Vite proxy):**
- Vite dev server proxies `/api/*` → `http://localhost:3000` so the browser sees a single origin
- Cookies use `SameSite=Lax`, `HttpOnly`, no `Secure` flag in dev (so localhost works)
- No CORS config needed locally because of the proxy

**Production (cross-origin):**
- Frontend on Vercel domain, API on Fly/Railway domain → real CORS
- `Access-Control-Allow-Origin` set to the frontend domain, `Allow-Credentials: true`
- Cookies switch to `SameSite=None; Secure`
- One `CORS_ORIGIN` env var on the API picks the allowed origin per environment

**Env var scaffolding** (`.env.example` checked in, `.env.local` gitignored):

```
# DB
DATABASE_URL=                # libsql://… (Turso) or file:./local.db (dev)

# Storage
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET=sojourn-assets
R2_PUBLIC_BASE_URL=          # https://assets.sojourn.app or pub-…r2.dev

# Stub source (Phase 0)
STUB_SOURCE=local            # local | r2

# Auth & app
EDIT_KEY_PEPPER=             # used when hashing edit keys
CORS_ORIGIN=http://localhost:5173
MIGRATE_ON_BOOT=1            # run drizzle migrations on Node startup; off in CI

# Phase 1+ placeholders (unused in Phase 0 but documented now)
PIXELLAB_API_KEY=
GEMINI_API_KEY=
ELEVENLABS_API_KEY=
```

**Migrations:** Drizzle migrations run on Node startup when `MIGRATE_ON_BOOT=1`. CI runs them as a separate step against a temp DB. Production: same boot-time migration is fine until we have multiple instances; revisit if/when we scale horizontally.

**Deploy sequencing:** picking Fly vs Railway is deferred until the first deploy actually happens — committing to one now is theater. Whichever wins, the contract is: `pnpm --filter api build && pnpm --filter api start`, with `MIGRATE_ON_BOOT=1` so the schema is current before the server accepts traffic.

## What's *not* in Phase 0 (and why)

- Real PixelLab calls (sprites) → Phase 1
- Real nano banana calls (portraits) → Phase 1
- Real ElevenLabs → Phase 2
- Lipsync of any kind → Phase 2
- AI-assist field actually working → Phase 2+
- Reference-image upload UX (drop zone, preview) → Phase 1, paired with nano banana
- Public gallery / discovery → Phase 3
